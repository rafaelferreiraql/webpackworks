The helper tools

Less preprocessor

Less takes (almost) everything that you miss in CSS and implements it. You can use nesting, variables, conditionals, and much more. Take, for example, the code used for the navigation buttons:

nav {
    background-color: white;

    ul {
        display: block;

        @media (min-width: 700px) {
            padding-left: .5em;

            @media (min-width: 800px) {
                padding-right: 12em;
            }
        }

        li {
            display: inline;
            margin-bottom: 1.2em;
            margin-right: .5em;
            text-align: center;

            list-style: none;

            a {
                // Item appearance
                color: white;
                background-color: @wpblue2;
                border-radius: 4px;
                padding: 8px;
                font-family: "Segoe UI",sans-serif;
                font-weight: bold;
                font-size: 1.5em;
                text-decoration: none;
                &:hover {
                    background-color: black;
                    color: @wpblue1;
                    transition: all linear .2s;
                }

                &.mainTab {
                    background-color: @mainred;
                    &:hover {
                        background-color: black;
                        color: @mainred;
                    }
                }

            }
        }
    }
}

Nesting means you won't have to write ul {} and ul li {} and ul li a {}, you can just nest them in a logical manner as ul { li { a { } } }. You can also see things like @wpblue1, which are variables (defined in another Less file).

Autoprefixer

When experimental CSS features were implemented in a web browser, the properties were frequently included in the engine prefixed by the respective browser vendors, until it is standardized. Autoprefixer parses your CSS (thus, it's used AFTER Less) and automatically inserts vendor prefixed properties when adequate; that lets you enjoy some additional compatibility with legacy versions that used these properties when they were experimental (thus applying these vendor prefixes).

Autoprefixer is part of the PostCSS ecosystem, which is by itself a framework for many other styling-related add-ons.

Pug templates

Even more exciting is Pug. First, take a look at how the files are organized:

|_ views
  |_ default.pug
  |_ home.pug
  |_ process.pug
  |_ tools.pug
  |_ webpack.pug

As HTML imports are, amazingly, not a thing yet between browsers, it's reassuring that you can use Pug templates for composition. Here, default.pug contains all (soon to be) HTML shared by all pages, and the other template files are the individual content.

That's not the only great thing about Pug. Have a look at the default.pug file and understand how its syntatical sugar is enough to make you diabetic:

doctype html
html
  head
    title Webpack Works!
    link(rel="stylesheet" href="styles.css")
  body
    .container
      header: .headcontent #[img(src="assets/wpicon.png" width="150px" height="150px")] #[span Webpack Works!]
      nav
        ul
          li: a(href="home.html") Home
          li: a(href="webpack.html") Web..what?
          li: a(href="tools.html") The tools
          li: a(class="mainTab" href="process.html") The process
      main
        block content
      footer
        p 2017 Rafael Ferreira

block content is where the individual content is inserted.

Other possibilities

So, that's all I used in this page, but depending on what you need, there's no shortage of various tools to be used in your project. For example, if I used Javascript, I could use loaders for Typescript or Coffeescript, as well as Babel for polyfilling; if I used more of PostCSS, I could be using maybe CSSNext, which polyfills future CSS specifications.
